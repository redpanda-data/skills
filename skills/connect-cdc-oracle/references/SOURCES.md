# connect-cdc-oracle Skill Source Map

Maps each file in `skills/connect-cdc-oracle/` to the source paths it derives from, so future
syncs and human maintainers know exactly where to verify claims.

The skill documents the `oracledb_cdc` **input** of Redpanda Connect (LogMiner-based CDC from
Oracle Database). Two public repos are the ground truth:

- **`redpanda-data/connect`** (Go) ā€” the connector implementation under `internal/impl/oracledb/`.
  This is an **enterprise** connector (Redpanda Community License; `license.CheckRunningEnterprise`),
  introduced in **4.83.0**. Connect is versioned by release **tag**; verify against a released tag,
  not `main`, when pinning behavior.
- **`redpanda-data/rp-connect-docs`** ā€” the connector reference page. The field list, types, and
  defaults are **auto-generated** into a partial from the Go config; overrides live in
  `docs-data/overrides.json`. Do **not** hardcode field defaults here ā€” defer to the generated partial.

Read both via the Redpanda-Github-Read MCP connector (`get_file_contents`) or `gh api .../contents/`.
Avoid `gh search code` (rate-limited). All connect paths verified on `redpanda-data/connect@main`;
docs paths on `redpanda-data/rp-connect-docs@main`.

## Resolved in this sync (release v4.99.0)

- **`snapshot_mode` vs `stream_snapshot` (RESOLVED 4.99.0, #4570):** `config.go` defines `SnapshotMode` with values `none`/`snapshot_only`/`snapshot_and_stream` (`parseSnapshotMode`); `input_oracledb_cdc.go` registers `snapshot_mode` as `NewStringEnumField(...).Version("4.99.0")` and marks `stream_snapshot` `.Deprecated()`. When `snapshot_mode` is unset, `stream_snapshot: true` maps to `snapshot_and_stream`, `false` to `none` (else `snapshot_mode` wins). The skill now documents `snapshot_mode` (incl. the new `snapshot_only` backfill mode) as the current field and `stream_snapshot` as the deprecated alias (`config-reference.md`, `pipeline-and-output.md`, `SKILL.md`, `resources/oracle-cdc-example.yaml`).
- **Snapshot metadata `scn` (4.98.0, #4542) + `commit_ts_ms` (4.99.0, #4571):** `replication/snapshot.go` `Prepare()` runs `SELECT CURRENT_SCN, SYSTIMESTAMP FROM V$DATABASE` and `processBatch` sets `MessageEvent.SCN`/`CommitTimestamp` on every `read` row; `batcher.go` `Publish()` emits `scn` (when `SCN.IsValid()`) and `commit_ts_ms` (when `CommitTimestamp` non-zero). So snapshot `read` messages now carry both (same value for all snapshot rows). The metadata tables in `SKILL.md` and `pipeline-and-output.md` were corrected from the prior "absent on snapshot" claim.
