# Redpanda Connect: Enterprise Features

Redpanda Connect ships a single binary that runs both Community (BSL-licensed, source-available) and Enterprise (RCL-licensed) components. Enterprise features are enabled by a valid Redpanda Enterprise Edition license. New installs get a 30-day trial; after the trial expires you are **blocked from using enterprise connectors** unless you upgrade to an Enterprise Edition license. Community components keep working.

This reference documents the Connect-domain enterprise features and their exact config keys, grounded in the Connect source and licensing docs. Each feature is marked with whether it **requires an Enterprise license**.

## Supplying a License

The license enables all enterprise functionality (enterprise connectors, etc.). Connect resolves a license from, in order:

```bash
# 1. CLI flag (highest precedence) — overrides env/file
redpanda-connect run --redpanda-license "YOUR_LICENSE_KEY" config.yaml
rpk connect run --redpanda-license "YOUR_LICENSE_KEY" config.yaml

# 2. REDPANDA_LICENSE env var (inline license string)
export REDPANDA_LICENSE="YOUR_LICENSE_KEY"

# 3. REDPANDA_LICENSE_FILEPATH env var (path to a license file)
export REDPANDA_LICENSE_FILEPATH=/path/to/redpanda.license

# 4. Default file path, applied automatically if present:
#    /etc/redpanda/redpanda.license
```

The `--redpanda-license` flag usage string: "Provide an explicit Redpanda License, which enables enterprise functionality. By default licenses found at the path `/etc/redpanda/redpanda.license` are applied."

License error at startup (when an enterprise component is used without a license):
`this feature requires a valid Redpanda Enterprise Edition license that includes the Connect product. For more information check out: https://docs.redpanda.com/redpanda-connect/get-started/licensing/`.

## Connect Enterprise Feature Summary

| Feature | Requires Enterprise License | What it gates |
|---|---|---|
| Enterprise connectors (CDC inputs, Snowflake/BigQuery/Iceberg/Splunk/OTLP/Slack/Drive/Salesforce — full list in [connector-catalog.md](connector-catalog.md)) | **Yes** — blocked after trial | Additional inputs/outputs/processors |
| Allow or deny lists (`connector_list.yaml`) | **No** (works without a valid license) | Restricts which components a pipeline may use |
| FIPS compliance | **No** (no change on expiry) | FIPS-compliant `rpk`/binary build |
| Redpanda Connect configuration service (global `redpanda` block: logs/status topics) | **No** (no change on expiry) | Sends Connect logs/status events to a Redpanda topic |
| Secrets management (remote lookup, `--secrets`) | **No** (no change on expiry) | Resolve secret values from remote systems at runtime |

> Allow/deny lists, FIPS, the configuration service, and secrets management are RCL-licensed enterprise features but are not disabled when a license expires. The hard gate on license expiry is the **enterprise connectors** (the enterprise-tier components listed in [connector-catalog.md](connector-catalog.md); AI/ML processors are certified tier and NOT gated).

## Enterprise Connectors (license required)

Enterprise connectors are inputs, outputs, and processors available only with an Enterprise license. Discover them in the catalog with the `support=enterprise` filter: https://docs.redpanda.com/redpanda-connect/components/catalog/?support=enterprise

### AI/ML processors (certified — no license gate)

AI/ML inference processors are **certified** tier, not enterprise: at Connect
v4.99.0 all 16 are marked `certified` in the component catalog
(`internal/plugins/info.csv`) and their source is Apache-2.0 with no runtime
license check. (This section previously called them enterprise connectors —
that was wrong.) They are documented here because they pair with the
license-gated connectors in AI pipelines. Verified processor names from the
component catalog:

- OpenAI: `openai_chat_completion`, `openai_embeddings`, `openai_image_generation`, `openai_speech`, `openai_transcription`, `openai_translation`
- AWS Bedrock: `aws_bedrock_chat`, `aws_bedrock_embeddings`
- Cohere: `cohere_chat`, `cohere_embeddings`, `cohere_rerank`

## Allow or Deny Lists (enterprise; not gated on expiry)

Limit which Connect components a Redpanda Connect instance is allowed to run. The file is read automatically from the fixed path:

```
/etc/redpanda/connector_list.yaml
```

Schema — exactly one of `allow` or `deny` (mutually exclusive; specifying both is an error):

```yaml
# Allow-list: ONLY these components may be used (everything else removed from the schema)
allow:
  - redpanda
  - mapping
  - http_server

# OR a deny-list: these components are removed; everything else is allowed
# deny:
#   - subprocess
#   - command
```

- `allow` and `deny` are arrays of component names. Providing both fails with: "connector list must only contain deny or allow items, not both".
- On startup Connect logs: `Successfully applied connectors allow/deny list from '/etc/redpanda/connector_list.yaml'`.
- An empty/missing file is a no-op (no restriction).

## Secrets Management (enterprise; not gated on expiry)

Resolve secret values from a remote system at runtime instead of plain env vars. Configure with the `--secrets` flag (repeatable / comma-listed URNs, attempted in order until a value is found). Default is `env:`.

```bash
# Default: environment-variable lookup only
redpanda-connect run config.yaml                       # implicit --secrets env:

# Look up from Redis, falling back to env
redpanda-connect run --secrets redis://localhost:6379 --secrets env: config.yaml

# Cloud secret managers
redpanda-connect run --secrets aws://<secret-arn-or-name>?role=<iam-role> config.yaml
redpanda-connect run --secrets gcp://<project>?audience=<aud> config.yaml
redpanda-connect run --secrets az://<vault-name> config.yaml

# Disable all secret lookups
redpanda-connect run --secrets none: config.yaml
```

Supported URN schemes (from the secrets resolver):

| Scheme | Source | Notes |
|---|---|---|
| `env:` | Environment variables | Default; the only entry unless overridden |
| `redis://` | Redis | e.g. `redis://host:port` |
| `aws://` | AWS Secrets Manager | `?role=` for assume-role; `?trim_prefix=` to strip a key prefix |
| `gcp://` | GCP Secret Manager | `?audience=`; `?trim_prefix=` |
| `az://` | Azure Key Vault | host is the vault name (`https://<host>`) |
| `none:` | Disabled | Returns no secrets |

In the config, reference a secret like an env var (`${SECRET_NAME}`); the configured secret lookups resolve it at runtime. Multiple `--secrets` entries form a tiered lookup attempted in order.

## FIPS Compliance (enterprise; not gated on expiry)

Run Connect with a FIPS-compliant build of `rpk`/the Connect binary to meet FIPS cryptography requirements. This is a build/runtime distribution concern (FIPS-validated crypto), not a YAML config field. See the rpk FIPS quickstart for installing the FIPS build. License behavior: no change on expiration.

## Redpanda Connect Configuration Service — global `redpanda` block (enterprise; not gated on expiry)

The top-level `redpanda` namespace can send Connect's own **logs** and **status events** to a topic on a Redpanda cluster (in addition to the standard logger). Relevant nested keys (full producer tuning is in [config-structure.md](config-structure.md)):

```yaml
redpanda:
  seed_brokers: ["localhost:9092"]   # required when using this block
  pipeline_id: ""                     # identifies this Connect instance in emitted events
  logs_topic: ""                      # topic to publish Connect logs to (empty = disabled)
  logs_level: info
  status_topic: ""                    # topic to publish lifecycle/status events to
  tls: { enabled: false }
  sasl: []
```

This block doubles as the shared connection config for the `redpanda` input/output (when their `seed_brokers` is omitted).
