# microsoft_sql_server_cdc: Config Reference

All config fields for the `microsoft_sql_server_cdc` input, grounded in
`connect/internal/impl/mssqlserver/input_mssqlserver_cdc.go` and the
corresponding autogenerated doc at
`connect/docs/modules/components/pages/inputs/microsoft_sql_server_cdc.adoc`.

## Complete Config Block (all fields with defaults)

```yaml
input:
  label: ""
  microsoft_sql_server_cdc:
    connection_string: ""              # required
    stream_snapshot: false
    max_parallel_snapshot_tables: 1
    snapshot_max_batch_size: 1000
    include: []                        # required (at least one pattern)
    exclude: []                        # optional
    checkpoint_cache: ""               # optional; omit to use built-in SQL Server cache
    checkpoint_cache_table_name: rpcn.CdcCheckpointCache
    checkpoint_cache_connection_string: "" # optional
    checkpoint_cache_key: microsoft_sql_server_cdc
    checkpoint_limit: 1024
    stream_backoff_interval: 5s
    auto_replay_nacks: true
    batching:
      count: 0
      byte_size: 0
      period: ""
      check: ""
      processors: []
```

## Field Reference

### `connection_string`

- **Type:** `string`
- **Required:** yes
- **Default:** none

The go-mssqldb / ADO.NET connection string for the source SQL Server database.

```yaml
connection_string: "sqlserver://connect_user:secret@sqlserver-host/instance?database=MyDatabase"
```

**Connection string format options:**

| Format | Example |
|---|---|
| DSN URL | `sqlserver://user:pass@host/instance?database=DB` |
| DSN URL (named instance) | `sqlserver://user:pass@host\SQLEXPRESS?database=DB` |
| DSN URL (port) | `sqlserver://user:pass@host:1433?database=DB` |
| ADO.NET style | `server=host;user id=user;password=pass;database=DB` |

The underlying driver is `github.com/microsoft/go-mssqldb`. The connection
string is passed directly to `sql.Open("mssql", ...)`. Common query parameters:

- `database=<name>` — target database
- `encrypt=true` — enforce TLS
- `TrustServerCertificate=true` — skip certificate validation (dev/testing only)
- `connection timeout=30` — connection timeout in seconds

---

### `stream_snapshot`

- **Type:** `bool`
- **Required:** no
- **Default:** `false`

When `true`, the connector first snapshots all existing rows from included
tables before entering CDC streaming mode. Each snapshot row emits a message
with `operation: read`.

The snapshot uses `SNAPSHOT` isolation level transactions and keyset-pagination
(ordering by primary key, fetching `snapshot_max_batch_size` rows at a time).
Tables without a primary key cannot be snapshotted — the pipeline errors at
startup if such a table is matched by `include`.

If a checkpoint already exists in the cache (pipeline restart), the snapshot
is **skipped** regardless of this setting, and streaming resumes from the
saved LSN.

```yaml
stream_snapshot: true
```

---

### `max_parallel_snapshot_tables`

- **Type:** `int`
- **Required:** no
- **Default:** `1`

Number of tables snapshotted concurrently during the snapshot phase. Increase
to speed up initial load when multiple large tables are included. Each parallel
worker opens its own database connection.

```yaml
max_parallel_snapshot_tables: 4
```

---

### `snapshot_max_batch_size`

- **Type:** `int`
- **Required:** no
- **Default:** `1000`

Maximum number of rows fetched per batch during the snapshot phase. Each batch
becomes one or more messages depending on `batching`. Larger values reduce
round-trips but increase memory usage.

```yaml
snapshot_max_batch_size: 5000
```

---

### `include`

- **Type:** `array` of `string` (regular expressions)
- **Required:** yes
- **Default:** none

Regular expressions for tables to include. Patterns are matched as **unanchored substrings** (using `regexp.MatchString`) against the fully-qualified table name in `schema.tablename` format. At least one pattern is required.

Because matching is unanchored, `dbo\.orders` also matches `dbo.orders_archive` and `dbo.orders_2024`. To match exactly, anchor the pattern with `^` and `$`.

The connector queries `sys.tables` + `sys.schemas` and applies both `include`
and `exclude` filters. After filtering, it verifies that each matched table has
a corresponding capture instance in `cdc.change_tables`. If any matched table
lacks a capture instance, the pipeline errors at startup.

```yaml
include:
  - "^dbo\\.orders$"        # exact match for dbo.orders only (anchored)
  - "dbo\\.customer.*"      # all tables starting with customer in dbo schema
  - "^sales\\..+"           # all tables in the sales schema (anchored prefix)
```

> Note: In YAML, use `\\.` to match a literal dot (the backslash escapes the
> dot in the regex, and the YAML double-backslash becomes a single backslash).

---

### `exclude`

- **Type:** `array` of `string` (regular expressions)
- **Required:** no
- **Default:** `[]`

Regular expressions for tables to exclude. Applied after `include`. A table
matching any `exclude` pattern is skipped even if it also matches `include`.

```yaml
exclude:
  - "dbo\\.audit_.*"    # also unanchored — use ^dbo\\.audit_.*$ to prevent partial matches
  - "dbo\\.temp_.*"
```

---

### `checkpoint_cache`

- **Type:** `string`
- **Required:** no
- **Default:** none (empty string)

The label of a Connect [cache resource](https://www.docs.redpanda.com/redpanda-connect/components/caches/about)
to use for storing the last successfully delivered LSN. This allows the pipeline
to resume from that position on restart.

When **not set**, the connector uses its built-in SQL Server-based checkpoint
cache, automatically creating a table and stored procedure in the source
database. See `checkpoint_cache_table_name` for details.

When set, the named cache resource must be declared in `cache_resources:` and
the Connect user no longer needs `CREATE TABLE`/`CREATE PROCEDURE` rights.

```yaml
cache_resources:
  - label: checkpoint_redis
    redis:
      url: redis://localhost:6379

input:
  microsoft_sql_server_cdc:
    checkpoint_cache: checkpoint_redis
    checkpoint_cache_key: mssql_mydb_cdc
    # ...
```

Supported cache types include `redis`, `memory`, `file`, `sql_raw`, and others.
For production use, prefer a persistent cache (redis, sql_raw) over `memory`.

---

### `checkpoint_cache_table_name`

- **Type:** `string`
- **Required:** no
- **Default:** `rpcn.CdcCheckpointCache`

The fully-qualified `schema.tablename` for the built-in SQL Server checkpoint
cache table. Only used when `checkpoint_cache` is not set.

The value must be in **exactly** `schema.table` format (two dot-separated parts). A fully-qualified three-part name like `database.schema.table` is rejected at startup with: `table name must be in the format schema.tablename`.

At startup, the connector:
1. Validates the table name (must be exactly two dot-separated parts; each part must be 1-128 characters matching `^[A-Za-z_][A-Za-z0-9_$]{0,127}$`).
2. Creates the stored procedure `<schema>.CdcCheckpointCacheUpdate` first (using `CREATE OR ALTER PROCEDURE`) for upsert operations.
3. Creates the cache table if it does not already exist:
   ```sql
   CREATE TABLE <schema>.<table> (
     cache_key varchar(7) NOT NULL PRIMARY KEY,
     cache_val varchar(100)
   );
   ```

**The schema must already exist.** The pipeline does not create the schema.
If using the default `rpcn` schema, create it manually:
```sql
CREATE SCHEMA rpcn;
```
Then grant the Connect user rights to create objects in it:
```sql
GRANT CREATE TABLE TO connect_user;
GRANT CREATE PROCEDURE TO connect_user;
```

```yaml
checkpoint_cache_table_name: rpcn.CdcCheckpointCache  # default
# or use a custom schema/table:
checkpoint_cache_table_name: dbo.connect_checkpoints
```

---

### `checkpoint_cache_connection_string`

- **Type:** `string`
- **Required:** no
- **Default:** none

Optional separate DSN for the database that stores the checkpoint cache table.
When set, the checkpoint table is created on this remote server instead of the
source database. Useful when:

- The Connect user does not have DDL rights on the source database.
- You want checkpoints stored on a separate, always-available SQL Server
  instance (e.g. a dedicated management DB).
- The source is read-only.

If `checkpoint_cache` is also set, that takes precedence and this field is
ignored.

```yaml
checkpoint_cache_connection_string: "sqlserver://admin:pass@mgmt-host/Checkpoints?database=Checkpoints"
```

---

### `checkpoint_cache_key`

- **Type:** `string`
- **Required:** no
- **Default:** `microsoft_sql_server_cdc`

The key under which the LSN checkpoint is stored in the cache. This field **only applies when `checkpoint_cache` is set** (an external cache resource). The built-in SQL Server cache ignores this field entirely — it always uses the fixed internal key `'max_lsn'`. Two pipelines pointed at the same built-in cache table will therefore collide on the single `max_lsn` row regardless of `checkpoint_cache_key`.

To isolate multiple pipelines when using the **built-in** SQL Server cache, use distinct `checkpoint_cache_table_name` values (or distinct schemas) — not this field.

```yaml
# Correct isolation with built-in cache: use distinct table names
# Pipeline A
checkpoint_cache_table_name: rpcn.CdcCheckpointCacheOrders

# Pipeline B
checkpoint_cache_table_name: rpcn.CdcCheckpointCacheCustomers
```

```yaml
# Isolation with external cache: use distinct keys
# Pipeline A
checkpoint_cache: my_redis
checkpoint_cache_key: mssql_pipeline_orders

# Pipeline B (same external cache resource)
checkpoint_cache: my_redis
checkpoint_cache_key: mssql_pipeline_customers
```

---

### `checkpoint_limit`

- **Type:** `int`
- **Required:** no
- **Default:** `1024`

Maximum number of messages that can be in-flight (read but not yet
acknowledged) at any given time. A higher value enables more output
concurrency and batching but uses more memory.

An LSN is only committed to the checkpoint cache after **all** messages at or
below that LSN have been acknowledged. This preserves at-least-once delivery.
If the pipeline crashes with 1024 in-flight messages, up to 1024 messages may
be re-delivered on restart.

```yaml
checkpoint_limit: 4096   # higher parallelism
checkpoint_limit: 256    # lower memory usage
```

---

### `stream_backoff_interval`

- **Type:** `duration` string
- **Required:** no
- **Default:** `5s`

How long the connector waits before re-polling the change tables after
exhausting all available changes. Reducing this decreases latency; increasing
it reduces SQL Server load for low-traffic tables.

```yaml
stream_backoff_interval: 5s   # default — good for high-traffic tables
stream_backoff_interval: 30s  # lower SQL Server load for infrequent tables
stream_backoff_interval: 1m   # minimal polling for near-static tables
```

---

### `auto_replay_nacks`

- **Type:** `bool`
- **Required:** no
- **Default:** `true`

When `true`, messages that are rejected (nacked) at the output level are
automatically replayed indefinitely, creating backpressure. When `false`,
nacked messages are dropped. Disabling auto-replay reduces memory usage on
high-throughput pipelines where original message data can be discarded
immediately after consumption.

---

### `batching`

- **Type:** `object`
- **Required:** no

A batching policy that controls how messages are grouped before being sent to
the output. Batching is applied at the input level (after CDC events are read).

| Sub-field | Type | Default | Description |
|---|---|---|---|
| `count` | int | `0` | Flush when this many messages are buffered (0 = disabled) |
| `byte_size` | int | `0` | Flush when total size reaches this many bytes (0 = disabled) |
| `period` | string | `""` | Flush after this duration even if `count`/`byte_size` not reached |
| `check` | string | `""` | Bloblang query; flush when it returns `true` |
| `processors` | array | `[]` | Processors applied to each completed batch |

**Important:** when no batching policy is configured (all fields at their zero defaults, making the policy a no-op), the input code forces `policy.Count = 1`, yielding **one message per batch**. To actually batch, set at least one of `count`, `byte_size`, or `period`.

```yaml
batching:
  count: 100
  period: 1s

# Flush every transaction boundary
batching:
  check: 'meta("operation") == "update_after" || meta("operation") == "insert" || meta("operation") == "delete"'
  period: 500ms
```

## Built-in Checkpoint Cache Internals

When `checkpoint_cache` is not set, the connector creates — in this order:

**1. Stored procedure** (`rpcn.CdcCheckpointCacheUpdate`, created first):
```sql
CREATE OR ALTER PROCEDURE rpcn.CdcCheckpointCacheUpdate
  @Key varchar(7),
  @Value varchar(100)
AS
BEGIN
  SET NOCOUNT ON;
  IF EXISTS (SELECT 1 FROM rpcn.CdcCheckpointCache WHERE cache_key = @Key)
    UPDATE rpcn.CdcCheckpointCache SET cache_val = @Value WHERE cache_key = @Key;
  ELSE
    INSERT INTO rpcn.CdcCheckpointCache (cache_key, cache_val) VALUES (@Key, @Value);
END;
```

**2. Table** (`rpcn.CdcCheckpointCache` by default, created if absent):
```sql
CREATE TABLE rpcn.CdcCheckpointCache (
  cache_key varchar(7) NOT NULL PRIMARY KEY,
  cache_val varchar(100)
);
```

The connector always stores a single row with `cache_key = 'max_lsn'` and
`cache_val` holding the serialized LSN value. The `checkpoint_cache_key` config field is ignored for the built-in SQL Server cache — the key is always `'max_lsn'`. Two pipelines sharing the same built-in table will overwrite each other's checkpoint; use distinct `checkpoint_cache_table_name` values to isolate them.

The procedure is used for the checkpoint `Set` operation. The `Get` operation
(at startup only) runs a direct `SELECT` against the table. This avoids the
limitation of the generic `cache_sql` Connect component, which does not support
SQL Server's upsert semantics.
